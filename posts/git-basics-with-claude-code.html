Why git matters when you use Claude Code

Claude Code can write and change files quickly. That power is only safe when you can undo mistakes and see exactly what changed. Git gives you both. This guide covers the five commands you need every day—nothing more.

What git actually does

Think of git as a photo album for your project. Every time you run git commit, you take a snapshot of your files. If something breaks later, you flip back to an earlier photo. Claude Code generates changes fast, so taking snapshots often is the safest habit you can build.

Set up git once

git config --global user.name "Your Name"
git config --global user.email "you@example.com"

You only need to do this one time per computer. Then, inside any project folder:

git init

The five commands you need daily

1. Check what changed

git status

Run this before and after every Claude Code session. It shows which files were added, changed, or deleted. Read it carefully—it only takes a few seconds and prevents surprises.

2. See the actual changes

git diff

Shows the line-by-line differences. Lines starting with + were added; lines starting with - were removed. Read this before committing so you know exactly what Claude Code changed.

3. Stage files for a snapshot

git add filename.py
# or stage everything changed:
git add .

Staging means "include this in the next snapshot." Use git add . when you trust all the changes shown by git diff.

4. Take the snapshot (commit)

git commit -m "Add login function"

Write a short message describing what you did, not how. Bad: "changes". Good: "Fix off-by-one error in score counter".

5. View your snapshot history

git log --oneline

Shows a compact list of all your commits. Each line has a short ID and your message.

Undo the last commit (safely)

If you committed something wrong and haven't shared it yet:

git reset HEAD~1

This un-commits the last snapshot but keeps your files unchanged. You can then fix things and commit again.

A safe workflow with Claude Code

  1. Before asking Claude Code to change anything: git status → confirm you are starting clean.
  2. Ask Claude Code to make one small change.
  3. Run git diff and read what changed.
  4. Test the change manually (run the program, open the file, check the output).
  5. If it looks good: git add . then git commit -m "short message".
  6. If it's wrong: fix it or ask Claude Code to revert, then commit again.

Small commits protect you. If step 5 of a 20-step session breaks something, you only need to undo one commit, not redo everything from scratch.

Quick reference card

  • git status — what's changed right now?
  • git diff — show me the exact changes
  • git add . — stage all changes
  • git commit -m "message" — take a snapshot
  • git log --oneline — see all snapshots
  • git reset HEAD~1 — undo the last snapshot (keep files)

Practice exercise

  1. Create a new folder and run git init inside it.
  2. Create a file called notes.txt and write one sentence in it.
  3. Run git add . and git commit -m "Add notes file".
  4. Open notes.txt, add a second sentence, save it.
  5. Run git diff — you should see your new sentence with a + in front.
  6. Commit again with git commit -m "Add second note".
  7. Run git log --oneline — you should see both commits.

Once this feels natural, you are ready to use Claude Code on real projects with confidence that you can always go back.
